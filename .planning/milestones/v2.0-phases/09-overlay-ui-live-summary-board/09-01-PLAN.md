# Plan 09-01: SummaryCardSchema + SummaryCardStore

## Goal

Add `SummaryCardSchema` to the shared Zod schema registry and create `SummaryCardStore` for reading and writing summary card rows in the `summary_cards` SQLCipher table.

## Context

`src/shared/schemas/index.ts` already holds all Zod schemas (the single source of truth). The `summary_cards` table DDL is already in the DB (created in Phase 6/7). This plan extends the schema file with the card schema and builds a typed DB wrapper so Plan 09-02 (SummaryCardTimer) can persist and push cards without repeating raw SQL.

## Tasks

### T1: Add SummaryCardSchema to shared schemas

**File:** `src/shared/schemas/index.ts`

**Action:** Modify

**Details:**

Read the full file first to find the correct insertion point. Append the following to the bottom of the file (after the last existing export), following the same pattern used for other schemas:

- Export `SummaryCardSchema` as a named `z.object()`:
  - `topic_headline`: `z.string()` — short noun phrase describing the main discussion topic
  - `key_points`: `z.array(z.string()).min(1).max(5)` — sentences directly reflecting what was said
  - `speaker_contributions`: `z.record(z.string(), z.string())` — keyed by speaker label, one sentence per speaker
- Export `SummaryCard` as `z.infer<typeof SummaryCardSchema>`
- Export `StoredSummaryCard` as a plain TypeScript interface (not a Zod schema) representing a fully hydrated DB row:
  ```
  id: string
  meeting_id: string
  card_index: number
  interval_start_seconds: number
  interval_end_seconds: number
  wall_time_label: string
  topic_headline: string
  key_points: string[]          // parsed from key_points_json
  action_items_mentioned: string[] // parsed from action_items_mentioned_json
  speaker_contributions: Record<string, string>  // parsed from speaker_contributions_json
  model_used: string
  generated_at: string
  created_at: number
  ```

Do not alter any existing schema. Do not reformat or reorganize the file beyond the addition.

### T2: Create SummaryCardStore

**File:** `src/main/store/SummaryCardStore.ts`

**Action:** Create

**Details:**

Create a class `SummaryCardStore` that wraps the `summary_cards` SQLite table. Import `Database` from `'better-sqlite3'` (the type import, consistent with how `src/main/store/db.ts` imports it — check that file for the exact import style before writing). Import `SummaryCard`, `StoredSummaryCard` from `'../../shared/schemas/index'`.

The constructor accepts `db: Database.Database` and prepares the three statements below as class fields using `this.db.prepare(...)` so they are compiled once.

Implement three public methods:

**`saveCard(meetingId: string, cardIndex: number, intervalStartSec: number, intervalEndSec: number, wallTimeLabel: string, card: SummaryCard, modelUsed: string): string`**
- Generate `id` via `crypto.randomUUID()`
- Compute `generated_at` as `new Date().toISOString()`
- Serialize `card.key_points` → `JSON.stringify(card.key_points)`
- Serialize `card.speaker_contributions` → `JSON.stringify(card.speaker_contributions)`
- `action_items_mentioned_json` → `'[]'` (SummaryCard schema does not include action items; the column exists in DDL for Phase 10 extensibility)
- INSERT the row with all DDL columns. Use `created_at` as `Date.now()` (milliseconds, integer, matching the DDL default convention).
- Return the generated `id`.

**`getCardsForMeeting(meetingId: string): StoredSummaryCard[]`**
- `SELECT * FROM summary_cards WHERE meeting_id = ? ORDER BY card_index ASC`
- Map each raw row to `StoredSummaryCard` by JSON-parsing the three `_json` columns.

**`getCardsSince(meetingId: string, sinceMs: number): StoredSummaryCard[]`**
- `SELECT * FROM summary_cards WHERE meeting_id = ? AND created_at > ? ORDER BY created_at DESC`
- Binds `[meetingId, sinceMs]`.
- Map each raw row to `StoredSummaryCard` identically to `getCardsForMeeting`.

Extract the row-mapping logic into a private method `private rowToStoredCard(row: Record<string, unknown>): StoredSummaryCard` to avoid repetition.

Do not expose the `db` instance publicly. Do not add methods beyond the three specified.
