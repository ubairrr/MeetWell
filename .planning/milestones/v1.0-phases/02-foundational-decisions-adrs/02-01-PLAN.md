    **Section 2 — Context / Problem Statement:**
    1–3 sentences: MeetingAssist captures live meeting audio (microphone + system audio) and produces transcripts and artifacts. Before any recording starts, the product must adopt a clear, fixed consent posture — the choice between disclosed/consent-gated recording and covert recording is an existential legal and ethical decision that gates every other product and architecture choice.

    **Section 3 — Decision Drivers:**
    List these drivers (from REQUIREMENTS.md DEC-01 definition and the ROADMAP phase goal):
    - Legal defensibility across all operating jurisdictions
    - User trust — the product's core value depends on trustworthiness
    - Ethical obligation to all meeting participants, not just the app user
    - Avoidance of existential liability (covert recording is a criminal offence in multiple jurisdictions)
    - Simplicity: one consent posture globally, no jurisdiction branching

    **Section 4 — Options Considered:**
    Use the 4-column options table from PATTERNS.md (| Option | Description | Rationale | Verdict |). Include the four options from PATTERNS.md §02-DEC-01:

    Option A — Disclosed + all-party consent gate (per-meeting):
    - Description: A consent dialog fires every time the user starts a recording session. The dialog contains a checkbox ("I confirm all meeting participants have been informed and consented to this recording.") and a Start Recording button that is disabled until the checkbox is checked. No checkbox → no capture.
    - Rationale: Highest-bar consent standard; legally defensible globally; no per-session ambiguity; consent is re-confirmed each meeting.
    - Verdict: **Chosen** (per D-02, D-03, D-04)

    Option B — Disclosed + one-party consent (jurisdiction-aware):
    - Description: Lower consent bar for jurisdictions where one-party consent is legal; app detects or lets user select jurisdiction.
    - Rationale: Reduces per-meeting friction for some users.
    - Verdict: Rejected — jurisdiction detection adds complexity; trust cost of any lower-bar mode outweighs the friction saving; increases liability surface.

    Option C — Setup-only consent acknowledgement:
    - Description: A one-time gate at first run; user acknowledges once and never sees a consent prompt again.
    - Rationale: Reduces ongoing friction.
    - Verdict: Rejected — no per-session accountability; user cannot confirm consent for each specific meeting's participants.

    Option D — Covert / undisclosed recording:
    - Description: No consent gate; app records without notifying participants.
    - Rationale: None acceptable.
    - Verdict: **Never ship.** Existential legal and ethical liability. The product is a disclosed, consent-first recorder. This is explicitly listed in the Out of Scope table of REQUIREMENTS.md.

    **Section 5 — Decision Outcome:**
    Open with the one-sentence locked verdict (per PATTERNS.md §"Posture statement"):
    "MeetingAssist is a disclosed, all-party-consent recorder: the app gates every recording session behind a per-meeting consent checkpoint and never conceals the fact of recording from any participant."

    Then document each sub-decision explicitly, citing the decision ID:

    - Per D-02: The consent gate fires per-meeting — a dialog appears every time the user starts a recording session, not as a one-time setup acknowledgement.
    - Per D-03: The gate mechanism is a checkbox + disabled Start Recording button. The checkbox text reads: "I confirm all meeting participants have been informed and consented to this recording." The Start Recording button is disabled until the checkbox is checked. No checkbox checked → no audio capture initiated.
    - Per D-04: All-party consent is required globally, with no jurisdiction exceptions. The app does not offer a lower-consent mode for one-party jurisdictions. Users in such jurisdictions may exceed the app's requirement but the app does not relax it.
    - Per D-05 (Content Protection — separate from consent posture): setContentProtection(true) is ON by default via Electron's setContentProtection API (→ macOS NSWindowSharingNone), making the overlay panel invisible in the user's own screen-share. This is a user-privacy feature — it hides the user's private workspace from their own screen-share audience. It is explicitly NOT a mechanism for concealing the fact that recording is happening. The ADR separates these two uses: content protection (keep, default-on) vs. concealing the fact of recording (never ship). Reference: [01-DNA-CATALOGUE.md §Technique 5](./../01-dna-deep-dive-project-setup/01-DNA-CATALOGUE.md) for the setContentProtection technical substrate.

    **Section 6 — Consequences:**
    Positive:
    - Legal defensibility: per-meeting confirmation creates an auditable consent moment for every recording.
    - User trust: the product's trustworthiness claim is grounded in a verifiable consent posture.
    - Clear ethical posture: all participants are treated as having rights, not just the app user.
    - Content protection default-on: user's overlay panel is not accidentally visible in screen-shares.

    Negative / Tradeoffs:
    - Per-meeting friction: users must confirm consent every session. Accepted — this friction is the cost of the trustworthy posture and is core to the product's value.
    - No lower-consent convenience mode: one-party-jurisdiction users must still confirm all-party consent. Accepted — complexity and trust cost of tiered consent outweigh the convenience gain.

    **Section 7 — Open Dependencies:**
    DEC-01 itself has no unresolved vendor dependencies. Cross-reference DEC-02:
    - The data-handling side of recording posture (storage encryption, retention, raw audio) is governed by [DEC-02](./02-DEC-02-data-handling-privacy.md).
    - DEC-02 has an open dependency on RSCH-03 (vendor DPA / no-training terms); see that ADR for details.

    **Cross-reference style:** Use relative markdown links (e.g., `[DEC-02](./02-DEC-02-data-handling-privacy.md)`, `[REQUIREMENTS.md DEC-01](../../REQUIREMENTS.md)`) per PATTERNS.md §"Cross-reference linking convention".
