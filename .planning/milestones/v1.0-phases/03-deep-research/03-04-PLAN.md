    **## Overview**: This document specifies the cross-meeting memory data model for MeetingAssist. It uses sqlite-vec (vec0 virtual tables) living inside the same SQLCipher-encrypted database as transcript data. Embeddings are encrypted at rest automatically (inherits SQLCipher encryption). This design feeds Phase 4 AI-SPEC (ContextEngine architecture) and Phase 5 PRD (persistence layer spec).

    **## sqlite-vec Technical Foundation**: Document the key facts:
    - Version 0.1.9 on npm registry (verified)
    - Written in pure C, no external dependencies, runs anywhere SQLite runs
    - Vectors stored in vec0 virtual tables; supports float, int8, and binary vectors
    - KNN queries via MATCH operator with ORDER BY distance LIMIT k
    - Custom metadata columns supported (label, timestamp, etc.) for pre-filter before distance calculation
    - Transactional semantics — vector inserts/updates/deletes are atomic within SQLite transactions
    - Lives inside the same SQLCipher DB — inherits encryption automatically
    - CRITICAL: Must be loaded via db.loadExtension(sqliteVecPath) before any vec0 table operations; path must be resolved from asar-unpacked location (see Common Pitfall 4)

    **## Recommended Schema**: Include the full SQL DDL for all five tables:
    1. meetings table (id TEXT PK, title, started_at INTEGER, ended_at INTEGER, participant_count, raw_audio_path TEXT NULL, created_at)
    2. transcript_segments table (id TEXT PK, meeting_id FK→meetings ON DELETE CASCADE, speaker_label, channel TEXT "mic"|"system", timestamp_start REAL, timestamp_end REAL, text TEXT, is_speech_final INTEGER DEFAULT 1, created_at)
    3. vec_chunks virtual table using vec0 (embedding float[1536], +chunk_id TEXT, +meeting_id TEXT, +speaker_label TEXT, +timestamp_start REAL, +text_preview TEXT)
    4. artifacts table (id TEXT PK, meeting_id FK→meetings ON DELETE CASCADE, artifact_type TEXT "mom"|"summary"|"key_points"|"action_items"|"dates", content_json TEXT, model_used TEXT, created_at)
    5. action_items table (id TEXT PK, meeting_id FK→meetings ON DELETE CASCADE, description TEXT, assignee_label TEXT, due_date TEXT NULL, ics_exported_at INTEGER NULL, created_at)

    **## Chunking Strategy**: Document the recommended chunking parameters:
    - Chunk unit: contiguous transcript segments by speaker turn or topic shift
    - Chunk size: ~300–500 tokens per chunk (corresponds to ~3–5 transcript_segments)
    - Overlap: 50-token overlap between adjacent chunks to prevent context split
    - Embedding dimension: 1536 (matches text-embedding-3-small from OpenAI; also matches Gemini embedding models — allows provider swap)
    - When to embed: after speech_final accumulates a chunk threshold OR at end-of-meeting batch pass
    - Privacy consideration: embeddings are derived from transcript text (not audio); they live in the same SQLCipher DB and are encrypted at rest

    **## KNN Query Pattern**: Include the SQL query example for finding top-5 transcript chunks most relevant to a query embedding (SELECT from vec_chunks JOIN meetings WHERE embedding MATCH ? ORDER BY distance LIMIT 5).

    **## In-Meeting vs Cross-Meeting Architecture**: Include the three-row comparison table:
    - In-meeting chat: Rolling transcript in LLM context (1M token window) — no embeddings needed; current meeting fits in context
    - End-of-meeting summary/MOM: Full transcript → LLM batch — single pass over complete meeting
    - Cross-meeting recall: sqlite-vec KNN + transcript_segments fetch — past meetings don't fit in context; semantic search surfaces relevant chunks

    **## Open Questions**: Document the two open design questions:
    1. Which embedding model should be the default? Recommendation: apply same baseURL adapter pattern to embeddings — embedding provider is configurable, not hardcoded
    2. Token counting for chunking: use OpenAI tiktoken or equivalent — do not hand-roll tokenizer (off-by-one errors cause silent retrieval failures)

    **## DEC-02 Compatibility**: Note that this schema is fully compatible with DEC-02: all data lives in SQLCipher DB (encryption at rest via safeStorage key), raw_audio_path is nullable (transcribe-then-delete-raw-audio posture), ON DELETE CASCADE ensures per-meeting delete works correctly.
