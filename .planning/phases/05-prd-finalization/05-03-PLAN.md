**Section 2: Guiding Strategy**
Two paragraphs:
1. First shippable unit (per D-19): The highest technical risk in MeetingAssist is the dual-channel audio capture pipeline. The RSCH-04 spike validated both paths on an isolated test, but integrating audiotee + Deepgram Nova-3 dual-WebSocket into an Electron main process, with AudioWorklet mic bridge from the renderer, is the most complex plumbing in the system. Everything else (artifact extraction, overlay UI, context engine, break assist) depends on a working transcript stream. Validating capture and persistence first eliminates the largest uncertainty in the build milestone before investing effort in downstream features.
2. Dependency-first sequencing (per D-20): The phase sequence follows dependency arrows — no phase begins until its dependencies are shippable (can be run end-to-end on a real meeting). Each phase produces one "shippable unit" that can be tested before proceeding. The non-obvious constraint is that ArtifactPipeline (batch extraction) is sequenced before the Live Summary Board (Phase 4 overlay), because the summary board reuses the same LLM adapter + Zod output pattern as the pipeline. Building the batch pipeline first establishes the LLM → Zod → IPC → renderer pattern that the card generation then follows cleanly.

**Section 6: Constraints and Notes for the Build Planner**
A bulleted list of constraints the build planner must observe:
- Electron version: Re-verify and pin at build milestone start. CLAUDE.md recommends 41 LTS; the RSCH-04 spike ran on 42. Both are acceptable; pin to the current LTS at build time (per SETUP-03 posture). Do not pin to a version without re-verifying.
- mip_opt_out: true must be hardcoded at SDK initialization in Phase 2. It must not be a user setting. This is a product commitment (DEC-02).
- Gemini free tier: The build milestone must include a settings validator that disqualifies Gemini free-tier keys. This must be implemented in Phase 3 (first LLM call). See RSCH-03.
- Consent gate: The SessionManager FSM consent guard must be enforced in the Electron main process in Phase 1, even if the full capture is not wired yet. Phase 2 connects the full capture; the FSM guard must already be in place.
- asarUnpack: Both better-sqlite3-multiple-ciphers .node file and the audiotee Swift binary must be in asarUnpack from Phase 2 onward. Failing to do this causes module load errors in packaged builds.
- EpochCompressor source: Must read from transcript_segments, NOT summary_cards. This anti-pattern is explicitly documented in AI-SPEC §2.2 Pitfall 4. Phase 5 implementation must be verified against this constraint.
- Eval harness: The adversarial corpus (60 transcripts, 8 categories) should be seeded in Phase 3 as transcripts become available. Running eval for the first time in Phase 6 creates a last-minute risk. Seed the corpus incrementally from Phase 3 onward.
