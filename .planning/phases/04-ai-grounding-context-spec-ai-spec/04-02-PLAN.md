    **3.1 Purpose and Scope**
    Explain that the adversarial eval harness is a standalone test tooling (not a production component) that validates whether the faithfulness contract from Section 1 is actually enforced by the ArtifactPipeline implementation. State that the harness is a shipping gate — the build milestone's ArtifactPipeline cannot be declared done until the harness passes. State that v1 uses synthetic adversarial transcripts only (per CONTEXT.md deferred ideas — real recorded samples are a post-launch iteration).

    **3.2 Primary Metric: Citation-Grounded Faithfulness Score (CGFS)**
    Document the CGFS metric:
    - Formula: CGFS = (citation-verifiable items) / (total extracted items)
    - Citation-verifiable definition: quote_full appears verbatim (or near-verbatim, >= 90% token overlap) in the ground-truth transcript segment at the cited timestamp, AND the claim derived from the quote does not introduce information beyond what the quote states
    - Target threshold: CGFS >= 0.85 (85% of all extracted items must be citation-verifiable)
    - Note that this is a shipping gate, not advisory — a CGFS below 0.85 blocks the build milestone's ArtifactPipeline from being declared done
    - Explain the threshold rationale: adapted from RAGAS standard (>= 0.8 production standard); set at 0.85 for meeting artifacts where false action items are especially damaging to user trust

    **3.3 Secondary Metric: Extrinsic Hallucination Rate (EHR)**
    Document the EHR metric:
    - Formula: EHR = (items with no matching evidence in transcript) / (total extracted items)
    - Target threshold: EHR <= 0.05 (at most 5% of extracted items may be extrinsic hallucinations)
    - Also a shipping gate

    **3.4 Tertiary Metric: Intrinsic Distortion Rate (IDR)**
    Document the IDR metric as manual-eval-only:
    - Definition: proportion of items where evidence exists in transcript but claim distorts it (wrong assignee, mangled date, inverted decision, dropped condition)
    - Manual-only because automated detection of subtle distortions is unreliable
    - No numeric gate for v1; human reviewer flags IDR cases during corpus review

    **3.5 Passing Bar (Shipping Gate)**
    State the complete three-part passing bar explicitly in a highlighted block:
    - CGFS >= 0.85 across all categories
    - EHR <= 0.05 across all categories
    - No single adversarial category scoring below CGFS 0.75
    - All three conditions must be met simultaneously before the ArtifactPipeline is considered shippable

    **3.6 Adversarial Transcript Corpus (v1)**
    Include the full 60-transcript corpus composition table from RESEARCH.md (Standard team syncs x10, Action-item-dense x10, Date-heavy x10, High speaker count x5, Adversarial — fabrication bait x10, Adversarial — attribution bait x5, Adversarial — implicit inference traps x5, Short/no-content x5). Total: 60 synthetic transcripts.

    Document the corpus generation method (Claude's Discretion per 04-CONTEXT.md): LLM-as-generator pipeline — (1) generate scenario, (2) generate transcript, (3) generate ground-truth extraction, (4) inject adversarial content for adversarial categories, (5) human review for naturalness and ground-truth accuracy.

    Include the AdversarialTestCase TypeScript interface from RESEARCH.md verbatim (transcript_id, category, transcript, ground_truth with action_items/decisions/dates, adversarial_injections).

    **3.7 Harness Architecture**
    Reproduce the harness architecture diagram from RESEARCH.md (adversarial_corpus/ directory → harness.ts → per-case pipeline: run ArtifactPipeline → collect MeetingArtifactsSchema → citation verifier → ground-truth comparison → per-case CGFS/EHR → eval_report.json). Note that the harness is a standalone TypeScript script (eval/harness.ts) runnable via npx ts-node — not part of the Vitest unit test suite. Document the citation verifier fuzzy matching rule: 90% token-level overlap threshold for near-verbatim criterion.

    **3.8 When to Run the Harness**
    Document the four trigger conditions from RESEARCH.md: (1) before any change to the extraction prompt, (2) before any change to Zod schema that affects citation fields, (3) at the start of the build milestone (baseline score), (4) after each extraction-related PR in the build milestone.
