<objective>
Build the entire data-layer foundation for named speaker attribution: the `speaker_aliases` table, the read-time roster queries, the JSON-safe rename-propagation utilities, and the `SpeakerAliasStore` that ties them together inside one atomic transaction per rename batch. This is the layer everything else in Phase 12 (IPC handlers in 12-03, UI in 12-04) is built on top of.

Purpose: Per CONTEXT.md D-05/D-06/D-07, renames must propagate deterministically into every derived artifact without ever mutating `transcript_segments.speaker_label` (the immutable-transcript anti-pattern) and without corrupting JSON when a display name contains `"`, `\`, or `$`. Per D-05's own admission, Stage 2 generation never substitutes real names at generation time, so this find/replace pass is the only mechanism that makes renamed speakers appear anywhere.
Output: `speaker_aliases` DDL, `speakerRename.ts`, `SpeakerAliasStore.ts`, extended `TranscriptStore`, and full unit-test coverage for all of it.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| IPC payload (renderer, untrusted) -> SpeakerAliasStore SQL execution (main, trusted sink) | The label/name mapping ultimately originates from renderer-controlled input (validated upstream in 12-03); this store must defensively parameterize regardless |
| Application data -> encrypted SQLite file (data at rest) | speaker_aliases and all propagated columns live in the same SQLCipher-encrypted DB file as everything else — no new at-rest exposure |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-12-01 | Tampering | SpeakerAliasStore.applyRenames SQL | high | mitigate | All queries use better-sqlite3 prepared statements with bound parameters (`?` / `@named`) exclusively — never string-concatenate originalLabel/newName/meetingId into SQL text |
| T-12-02 | Tampering | speakerRename.ts renameInContentJson | high | mitigate | Always JSON.parse -> deep-walk-mutate decoded values -> JSON.stringify; never regex over raw serialized JSON text, which would corrupt the record if a display name contains `"`/`\`/newlines |
| T-12-03 | Information Disclosure | SpeakerAliasStore.applyRenames (all 5 propagation targets) | high | mitigate | Every SELECT/UPDATE touching artifacts/action_items/summary_cards/epoch_summaries/speaker_aliases is parameterized with `meeting_id = ?` — no query may omit this scope (SPKR-05) |
| T-12-04 | Denial of Service | speakerRename.ts regex compilation | low | accept | User-provided newName is only ever used as a replacement string, never compiled into a RegExp; only the small, enum-bounded original label ("You"/"Speaker N"/a prior alias) is ever compiled, and escapeRegExp neutralizes special characters first — ReDoS via pathological display name is not reachable |
</threat_model>
