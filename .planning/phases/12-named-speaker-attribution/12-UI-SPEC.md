# Phase 12 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none — hand-rolled React function components with inline `style={{...}}` props (no CSS-in-JS library, no Tailwind, no CSS Modules) |
| Icon library | none — Unicode glyphs only (`▼` `▶` `✓`); no icon-font/SVG-icon package in `package.json` |
| Font | `Inter, -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, ...` (declared in `src/renderer/src/assets/base.css`) |

**shadcn gate ruling (do not re-run this decision without a PRD update):** `components.json` and `tailwind.config.*` are both absent from the repo. The tech stack is Electron 41/42 + React 19 + Vite 7, which technically qualifies for the shadcn init prompt — but this is phase 12 of an already-shipped v2.0 codebase (11 phases, 59 plans built entirely on plain inline-style React components, confirmed by direct reads of `ArtifactItem.tsx`, `ArtifactReview.tsx`, `CitationPanel.tsx`, `App.tsx`, `CapturingScreen.tsx`). Introducing a component library / Tailwind now would be an unrequested architectural change into a project whose CLAUDE.md explicitly states "No new architectural decisions without updating the PRD docs first." This phase formalizes the **existing informal token system** already used consistently across the overlay instead of introducing shadcn. If a future milestone wants to adopt shadcn, that should be its own scoped decision, not smuggled in via a small propagation-logic phase.

### Existing component patterns to reuse (do not reinvent)
- `ArtifactItem.tsx` — `isEditing` / textarea / Save / Cancel inline-edit pattern. This is the direct visual precedent for each row's rename `<input>` in the new modal.
- `ArtifactReview.tsx`'s meeting-title save flow (`meetingTitle` / `titleSaved` state, `set-meeting-title` IPC) — direct precedent for a single-field-per-row "stage locally, commit on save" pattern, generalized to N rows in this phase.
- `CitationPanel.tsx` — precedent for a bordered/inset sub-panel look (`rgba(255,255,255,0.05)` fill, `borderLeft: 2px solid #4b5563`) usable for the excerpt line under each speaker row if a visually distinct excerpt treatment is wanted.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gaps, tight inline spacing (excerpt-to-input gap) |
| sm | 8px | Row internal padding, gap between staged-name input and Save/Cancel controls |
| md | 16px | Modal padding (left/right), section spacing between roster list and footer buttons |
| lg | 24px | Modal top/bottom outer padding, separation from the "Rename Speakers" trigger button to the rest of ArtifactReview's footer |

Exceptions: the shipped overlay (`ArtifactItem.tsx`, `CitationPanel.tsx`, `ArtifactReview.tsx`) already uses several sub-4px/off-scale values (`2px`, `6px`, `10px`) throughout, driven by the 380px overlay width forcing very tight information density. **This phase's new `RenameSpeakersModal.tsx` must hold the strict 4/8/16/24 scale for its own layout** (it is new surface, not a retrofit) — do not introduce new 2px/6px/10px values. It is acceptable for the modal to visually sit inside the existing 380px-wide overlay column, so it inherits that width constraint but not the off-scale spacing habits.

|------|------|--------|-------------|
| Body | 12px | 400 | 1.5 |
| Label | 11px | 600 | 1.4 |
| Heading | 14px | 600 | 1.2 |
| Display | — (not used in this phase; no display-scale text in a 380px overlay modal) | — | — |

Rationale: matches the existing overlay's established sizes exactly — `ArtifactItem` body text is 12px/400, `CitationPanel` speaker/meta labels are 11px, `ArtifactReview`'s meeting-title heading is 14px/600. Reusing these (rather than inventing new sizes) keeps the new modal visually indistinguishable in density from the screen it's launched from.

Only 3 sizes are needed for this phase (Display is unused) — do not add a 4th without updating this contract.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (trigger) | "Rename Speakers" — button in `ArtifactReview`, placed in the footer button group alongside "Start New Meeting" / "Dismiss", or as a small action inside the Action Items/Summary header row (executor's placement discretion per CONTEXT.md's "exact modal layout" discretion note); label text itself is locked |
| Primary CTA (commit, inside modal) | "Save Names" — not just "Save", to disambiguate from the meeting-title Save button already on the same screen |
| Secondary action (inside modal) | "Cancel" — discards staged edits, no confirmation required (non-destructive: nothing has been persisted yet) |
| Row input placeholder | "Enter name…" |
| Row excerpt fallback (no transcript text found for a label — should not normally occur per Pitfall in RESEARCH.md, but must degrade gracefully) | "(no transcript excerpt available)" |
| Empty state — no speaker labels detected for this meeting | Heading: "No speakers to rename" / Body: "This meeting has no transcript segments yet, so there's nothing to relabel." (should be effectively unreachable in practice — every completed meeting has at least the "You" label — but the modal must not crash or show a blank list if it happens) |
| Error state — save fails (IPC error / validation rejection) | "Couldn't save these names. Your changes weren't lost — try Save again." (mirrors the existing artifact-generation error banner's "your transcript is saved" reassurance pattern from `ArtifactReview.tsx`) |
| Error state — session not in `Complete` state (server-side gate rejection, should not be reachable via UI since the button only renders post-meeting, but must have a message if hit) | "Renaming is only available after the meeting ends." |
| Destructive confirmation | Not applicable — this phase has no destructive action. Renaming is reversible (can be renamed again) and Cancel from the modal discards only *unsaved, unpersisted* edits, which needs no confirmation dialog, consistent with `ArtifactItem.tsx`'s existing Cancel behavior (no confirm prompt there either) |

## Phase-Specific Notes for Planner/Executor

These supplement (do not replace) the sections above — included because this phase's UI surface is small enough that literal component/state inventory belongs directly in the contract.

**New component:** `src/renderer/src/components/RenameSpeakersModal.tsx`

**States the modal must render (all styled per the tokens above):**
1. **Loaded roster** — one row per distinct `speaker_label` (including "You"), each row: label chip/text (read-only, e.g. "Speaker 1"), representative excerpt (12px, italic, secondary-text color — reuse `CitationPanel`'s quote styling: `fontStyle: 'italic'`, `color: '#d1d5db'`), and a rename `<input>` pre-filled with the current effective name if one exists in `speaker_aliases`, else empty with the "Enter name…" placeholder.
2. **Staged-but-unsaved** — Save button uses accent color only once at least one row has a non-empty, changed value (disabled/neutral state otherwise, mirroring `ArtifactReview`'s title-save button exact disabled styling: `background: '#1f2937'`, `color: '#4b5563'`).
3. **Saving** — Save button shows a busy label (reuse the existing pattern from `ArtifactReview`'s export button: `"Exporting..."` → here, `"Saving..."`), disabled during the IPC round-trip.
4. **Save error** — inline error banner using the Destructive token, copy per Copywriting Contract above; modal stays open with staged edits intact so the user can retry without re-typing.
5. **Save success** — modal closes; the underlying `ArtifactReview` artifacts state is replaced with the IPC response payload (per RESEARCH.md Pitfall 6 — there is no separate "success toast" needed, the visible re-render of MOM/Summary/Key Points/Action Items with the new names *is* the confirmation).
6. **Empty roster** — see Copywriting Contract's empty-state row; Save button is not rendered (nothing to save).

**Layout constraint:** the modal renders inside/over a 380px-wide overlay window (`overlayWidth` default in `src/main/index.ts:42`). Do not design for a wider viewport — all row layouts (excerpt + input + label) must wrap or stack vertically within ~380px minus the `md` (16px) horizontal padding token, i.e. an effective content width of ~348px.
