# Plan 07-07: Main process wiring — replace stubs, FSM hooks, CaptureService instantiation

## Objective

Replace the two stub `ipcMain.handle` calls in `src/main/index.ts` with real implementations, instantiate `CaptureService` after the DB opens, and hook `CaptureService.startCapture` / `stopCapture` into the `SessionManager` state-change callback. After this plan, the full pipeline from consent → FSM transition → audio capture → Deepgram → DB is live.

## Context

**Current stubs to replace (from Phase 6 `src/main/index.ts` lines 88–89):**

```typescript
ipcMain.handle('mic-audio-chunk', () => undefined)
ipcMain.handle('end-meeting', () => undefined)
```

Both must become real implementations calling `CaptureService`.

**`SessionManager` hook:** The existing `session.onStateChange` callback in `index.ts` currently only sends `session-state-changed` to the renderer. Phase 7 extends it to also call `captureService.startCapture(meetingId)` on `Capturing` entry and `captureService.stopCapture()` on `Processing` entry.

**Meeting ID generation:** `CaptureService.startCapture` takes a `meetingId` string. Generate it with `crypto.randomUUID()` at the point of `Capturing` state entry (in the `onStateChange` hook). Store it in a local variable in the `app.whenReady()` closure so `stopCapture` can return it (the return value is not needed in Phase 7 but will be used in Phase 8).

**Deepgram API key:** Read from `process.env.DEEPGRAM_API_KEY`. If the env var is absent at startup, log a clear error and do not instantiate `CaptureService` — the app must still launch (the key may be entered via Settings UI in Phase 9), but `startCapture` will fail gracefully. For Phase 7 integration testing, the developer must set `DEEPGRAM_API_KEY` in a `.env` file at the project root (electron-vite loads `.env` automatically via `dotenv`).

**`ipcMain.handle` cannot be registered twice.** Phase 6 already registered `'mic-audio-chunk'` and `'end-meeting'`. Replace those stubs — do NOT add duplicate registrations. The correct pattern is to either:
- Remove the stub and add the real handler in the same edit, OR
- Use `ipcMain.removeHandler('channel')` before re-registering

The simplest approach: edit `index.ts` to replace both stub lines with real implementations in-place.

**`mic-audio-chunk` IPC handler note:** The `ipcMain.handle` callback receives `(event, payload)` where `payload` is the `ArrayBuffer` sent by the renderer. Pass it directly to `captureService.handleMicChunk(payload as ArrayBuffer)`.

**`end-meeting` IPC handler:** Call `session.transition('end-meeting')`. The FSM transition from `Capturing → Processing` fires `onStateChange`, which calls `captureService.stopCapture()`.

**Mouse event control:** When the session enters `Capturing`, the overlay must remain interactive so the user can click "Stop Meeting". Change `win.setIgnoreMouseEvents(false)` on `Capturing` entry and restore `win.setIgnoreMouseEvents(true, { forward: true })` on all other states. This is a direct addition to the `onStateChange` callback.

```typescript
import crypto from 'crypto'
```

Inside `app.whenReady().then(async () => { ... })`, after `const session = new SessionManager()`:

**Step 1 — Instantiate CaptureService:**

```typescript
const apiKey = process.env.DEEPGRAM_API_KEY ?? ''
if (!apiKey) {
  console.warn('[MeetingAssist] DEEPGRAM_API_KEY not set — capture will fail until key is configured')
}
const captureService = new CaptureService(db!, win, apiKey)
```

Note: `db` is of type `Database.Database | null`. It cannot be null here because `app.quit()` was called if `openDatabase()` failed. Use the non-null assertion `db!` or add a guard that returns early.

**Step 2 — Replace the `onStateChange` callback:** The current callback sends `session-state-changed`. Replace it entirely with an expanded version:

```typescript
let currentMeetingId: string | null = null

session.onStateChange((state, previous) => {
  if (win) {
    win.webContents.send('session-state-changed', { state, previous })
  }

  // Mouse event control: interactive during Capturing only
  if (win) {
    if (state === 'Capturing') {
      win.setIgnoreMouseEvents(false)
    } else {
      win.setIgnoreMouseEvents(true, { forward: true })
    }
  }

  // Capture lifecycle
  if (state === 'Capturing') {
    currentMeetingId = crypto.randomUUID()
    captureService.startCapture(currentMeetingId).catch((err: unknown) => {
      console.error('[MeetingAssist] CaptureService.startCapture failed:', err)
    })
  }

  if (state === 'Processing') {
    captureService.stopCapture().catch((err: unknown) => {
      console.error('[MeetingAssist] CaptureService.stopCapture failed:', err)
    })
    currentMeetingId = null
  }
})
```

**Step 3 — Replace the stub IPC handlers:**

Remove this line:
```typescript
ipcMain.handle('mic-audio-chunk', () => undefined)
```

Replace with:
```typescript
ipcMain.handle('mic-audio-chunk', (_event, buffer: ArrayBuffer) => {
  captureService.handleMicChunk(buffer)
})
```

Remove this line:
```typescript
ipcMain.handle('end-meeting', () => undefined)
```

Replace with:
```typescript
ipcMain.handle('end-meeting', () => {
  try {
    session.transition('end-meeting')
  } catch (err) {
    console.error('[MeetingAssist] end-meeting transition failed:', err)
  }
})
```

All other stub handlers (`start-break`, `end-break`, `confirm-artifact`, etc.) remain unchanged stubs — they are Phase 9 and Phase 8 concerns.

**Step 4 — Verify DB non-null before CaptureService:**

The existing code does:
```typescript
try {
  db = openDatabase()
} catch (err) {
  console.error('[MeetingAssist] DB init failed:', err)

  return
}
```

If `openDatabase()` throws, `app.quit()` + `return` means `CaptureService` instantiation is never reached — `db` is guaranteed non-null by the time we instantiate `CaptureService`. Use `db!` assertion.
